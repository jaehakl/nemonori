# Baseball Manager v1 개발 명세

## 1. Summary

- 이 문서는 `baseball-manager` 게임의 v1 구현 명세서다.
- 기술 방향은 `React 셸 + 순수 TypeScript 시뮬레이션 엔진 + SVG/DOM 기반 경기 UI`로 고정한다.
- v1 목표는 시즌 운영 중심의 프로야구 매니지먼트 시뮬레이션을 구축하는 것이다.
- 핵심 차별점은 `숨겨진 선수 능력치(Fog of War)`와 이를 간접 정보로 해석하는 운영 경험이다.
- v1에서 확정하는 핵심 축은 다음과 같다.
  - 10개 팀 가상 리그
  - 숨겨진 선수 능력치
  - 코치 리포트와 선수 면담 피드백
  - 1군/2군 운영과 수동 저장
  - 내부적으로는 투구 단위 경기 엔진
  - UI는 전광판 + 텍스트 중계 + 필드 마커 중심

## 2. Technical Direction

### 2.1 Framework

- 게임 페이지는 기존 앱 구조를 따르는 Next.js/React 클라이언트 컴포넌트로 구현한다.
- 게임 상태 표현은 React가 담당하되, 게임 규칙 계산은 React 밖의 순수 TypeScript 모듈에서 처리한다.
- Phaser, Babylon.js 같은 그래픽 엔진은 사용하지 않는다.

### 2.2 Rendering

- 메인 UI는 DOM 패널 기반으로 구성한다.
- 경기 화면의 필드는 SVG로 렌더링한다.
- 필드에서는 포지션 마커, 주자 상태, 수비 배치, 현재 타석 상황 정도만 시각화한다.
- 복잡한 실시간 애니메이션이나 물리 연출은 v1 범위에서 제외한다.

### 2.3 Save

- 저장은 기존 로컬 저장 프로토콜을 재사용한다.
- 기본 정책은 `수동 저장`이다.
- 저장 대상은 구단 운영 상태 전체와 시즌 진행 상태 전체다.
- 저장 슬롯은 게임 슬러그 단위로 관리한다.

## 3. Product Pillars

### 3.1 Fog of War

- 선수의 raw 능력치는 숫자로 직접 공개하지 않는다.
- 유저는 아래 공개 정보만으로 선수를 판단해야 한다.
  - 누적 성적
  - 최근 출장 기록
  - 선수 면담 코멘트
  - 코치진 텍스트 리포트
- UI는 어떤 화면에서도 hidden stat 숫자를 직접 표시하면 안 된다.

### 3.2 운영 중심 플레이

- 플레이의 중심은 경기 조작보다 시즌 운영이다.
- 유저의 핵심 결정은 다음과 같다.
  - 엔트리 구성
  - 선발 라인업 관리
  - 투수 운용과 휴식 분배
  - 1군/2군 승격 및 말소
  - 코치진 정보 해석
  - 드래프트 지명

### 3.3 정보의 불완전성

- 선수는 자신의 상태를 왜곡해서 보고할 수 있다.
- 코치진은 이를 교차 검증하는 역할을 한다.
- 코치 능력, 선수 멘탈, 최근 출장 이력이 함께 판단 재료가 된다.

## 4. Scope

### 4.1 v1 In

- 시즌 일정 진행
- 자동 경기 시뮬레이션
- 라인업/엔트리 관리
- 1군/2군 운영
- 코치 리포트
- 선수 면담 피드백
- 체력 회복과 혹사 리스크
- 2군 성장 처리
- 드래프트
- 경기 중 간단 개입
  - 투수 교체
  - 대타
  - 대주자
  - 대수비
  - 번트 등 기본 작전 지시

### 4.2 v1 Out

- Phaser/Babylon 기반 씬 렌더링
- 실시간 액션형 연출
- 복잡한 수비 위치 이동 애니메이션
- 투구별 구종/코스 수동 지정 UI
- 실존 팀/실존 선수 데이터 사용
- 자동 저장

## 5. Layered Architecture

### 5.1 Engine Layer

책임:

- 전체 게임 상태 보관
- 날짜 진행
- 유저 액션 처리
- 저장/불러오기
- 화면 전환 상태 관리

구성 예시:

- `createNewGame`
- `gameReducer`
- `selectors`
- `save/load adapter`

규칙:

- engine은 UI 프레임워크에 의존하지 않는다.
- engine은 외부에 `GameState`와 `UserAction` 중심 인터페이스를 제공한다.

### 5.2 Sim Layer

책임:

- 생성형 리그 구성
- 선수/코치 생성
- 피로와 회복
- 성장, 노쇠, 부상
- 경기 일정 처리
- 투구 단위 경기 시뮬레이션
- 드래프트 처리

구성 예시:

- `calendar`
- `player-growth`
- `player-condition`
- `coach-report`
- `game-sim`
- `pitch-sim`
- `roster`
- `draft`

규칙:

- sim은 순수 함수 또는 순수 함수에 가까운 계산 모듈로 작성한다.
- 동일한 seed와 동일 입력이면 동일 결과가 재현되어야 한다.

### 5.3 UI Layer

책임:

- 공개 정보 렌더링
- 패널별 인터랙션
- selector 결과 표시
- 경기 화면의 SVG/DOM 조합 렌더링

화면 구성:

- 대시보드
- 로스터/엔트리 화면
- 선수 상세 카드
- 경기 화면
- 드래프트 화면
- 저장/불러오기 패널

규칙:

- UI는 hidden stat 원본을 직접 읽지 않는다.
- UI는 selector가 제공하는 공개 정보만 소비한다.

## 6. League Model

### 6.1 League Structure

- 10개 팀 가상 리그를 사용한다.
- 팀당 144경기를 치른다.
- 상대 팀당 16경기 기준으로 운영한다.
- 정규 일정은 화요일부터 일요일까지 주 6경기 구조를 기본으로 한다.

### 6.2 Generated League

- 실존 데이터 대신 생성형 리그를 사용한다.
- 팀, 선수, 코치 모두 규칙 기반으로 생성한다.
- 생성 대상:
  - 팀명/팀 컬러/간단한 팀 성향
  - 선수 이름
  - 나이
  - 포지션
  - 잠재력
  - 성향
  - 코치진 특성

### 6.3 Reproducibility

- 새 게임 시작 시 RNG seed를 상태에 저장한다.
- 생성형 선수단과 경기 결과는 seed 기준으로 재현 가능해야 한다.

## 7. Player Model

### 7.1 Hidden Categories

선수 능력치는 다음 3개 묶음으로 관리한다.

- `physical`
- `technical`
- `mental`

### 7.2 Physical

- 신인 시점의 최고 잠재치가 사실상 상한 역할을 한다.
- 노쇠화와 혹사는 영구 하락을 유발할 수 있다.
- 최대 체력은 공통 프레임을 사용하되 회복력과 내구성은 선수별로 다르다.
- 예시 요소:
  - 회복력
  - 내구성
  - 근력
  - 주력

### 7.3 Technical

- 코치 역량의 영향을 가장 크게 받는 성장 영역이다.
- 투수 예시:
  - 구위
  - 제구력
  - 구종별 숙련도
  - 견제/수비
- 타자 예시:
  - 컨택
  - 선구안
  - 파워
  - 주루 센스
- 수비는 포지션별 숙련도로 관리한다.
- 유사 포지션 출전 시에도 숙련도가 일부 공유 상승한다.

### 7.4 Mental

- 경기 상황 경험에 따라 변한다.
- technical 성장률과 위기 대응 보정에 영향을 준다.
- 예시 요소:
  - 침착성
  - 투쟁심
  - 야구 지능
  - 성실성

### 7.5 Condition Rules

- 유저는 체력 숫자를 직접 보지 못한다.
- 컨디션 판단은 다음 정보의 조합으로 한다.
  - 최근 출장 이력
  - 투구 수/소화 이닝/포지션별 부담
  - 선수 면담 결과
  - 코치 리포트
- 출전 시 기본 체력 감소가 발생한다.
- 포수와 투수는 체력 부담이 크다.
- 지명타자는 체력 부담이 가장 낮다.
- 매일 일정량 회복한다.

## 8. Feedback and Report System

### 8.1 Player Interview

- 선수는 mental 성향에 따라 상태를 왜곡할 수 있다.
- 대표 패턴:
  - 열정/허세형: 괜찮다고 과대 보고
  - 엄살/소극형: 힘들다고 과소 보고
  - 객관/프로형: 비교적 정확한 보고

### 8.2 Coach Report

- 코치는 유저의 눈이자 육성 책임자다.
- 코치 능력이 높을수록 다음이 개선된다.
  - 텍스트 리포트 정확도
  - 선수 피드백 진위 판별
  - 담당 선수 성장 속도

### 8.3 Coach Structure

- v1 코치진은 기능형 구조로 둔다.
- 최소 구성:
  - 투수 담당
  - 타격 담당
  - 수비 담당
  - 멘탈 담당
- 스카우트/포지션별 세분화 코치는 후속 버전으로 미룬다.

## 9. Roster and Team Operations

### 9.1 Team State

- 팀은 1군/2군 로스터를 가진다.
- 1군 정원 제한을 둔다.
- 2군 강등 시 10일 후 콜업 가능 규칙을 둔다.
- 2군에서도 체력 회복과 성장 처리를 수행한다.

### 9.2 User Decisions

- 선발 라인업 제출
- 선발 투수 지정
- 불펜 운용
- 휴식일 관리
- 승격/말소
- 대체 선수 기용

### 9.3 Staff Influence

- 담당 코치 능력에 따라 2군 성장 속도와 리포트 질이 달라진다.

## 10. Match Simulation

### 10.1 Simulation Resolution

- 내부 엔진은 `투구 단위`로 설계한다.
- 각 투구는 다음 요소를 반영한다.
  - 투수 technical
  - 타자 technical
  - 양측 mental 보정
  - 체력/피로 보정
  - 현재 상황 보정
  - 작전 보정

### 10.2 UI Presentation

v1 경기 UI는 다음 3축으로 표현한다.

- 전광판
  - 이닝
  - 점수
  - BSO
  - 아웃카운트
  - 타순
- 텍스트 중계 패널
  - 투구 결과
  - 타격 결과
  - 주자 진루/득점
  - 교체와 작전 로그
- SVG 필드
  - 수비 포지션 마커
  - 주자 점유 상태
  - 현재 타자/투수 강조

### 10.3 Manual Intervention

- 기본은 자동 진행이다.
- v1 개입 범위:
  - 투수 교체
  - 대타
  - 대주자
  - 대수비
  - 번트 등 간단 작전
- 투구마다 구종/코스를 직접 입력하는 기능은 v1에 넣지 않는다.

## 11. Draft and Offseason

### 11.1 Draft

- 전년도 순위 역순으로 신인을 지명한다.
- 신인도 hidden stat 구조를 따른다.
- 유저는 아래 정보만 보고 판단한다.
  - 아마추어 누적 성적
  - 코치/스카우트성 의견 텍스트

### 11.2 Retired Player Conversion

- 은퇴 선수가 코치로 전환될 수 있다.
- 전환 후 코치 능력치는 현역 시절 hidden stat 기반으로 결정한다.
- 이 기능은 v1 설계 항목에 포함하되, 구현 우선순위는 드래프트보다 낮다.

## 12. Public Interfaces / Types

### 12.1 Core Types

문서 기준 최소 타입 계약:

- `GameState`
  - 현재 날짜
  - 리그 상태
  - 팀 상태
  - 현재 화면
  - 최근 로그
  - RNG seed
- `Player`
  - 공개 프로필
  - 비공개 hidden stat
  - 체력/피로 상태
  - 포지션 숙련도
  - 출장 이력
  - 부상/노쇠 상태
- `CoachReport`
  - 텍스트 리포트
  - 신뢰도
  - 추정 방향
  - 최근 갱신 시점
- `MatchState`
  - 이닝
  - 점수
  - 아웃
  - BSO
  - 주자 상태
  - 현재 타자/투수
  - 수비 배치
  - 교체 가능 상태
  - 로그
- `UserAction`
  - 라인업 수정
  - 엔트리 이동
  - 하루 진행
  - 경기 시작/계속 진행
  - 교체
  - 작전 지시
  - 저장/불러오기

### 12.2 Selector Contract

- selector는 공개 정보 전용 뷰 모델을 반환한다.
- selector가 반환하는 선수 카드 정보에는 다음만 포함한다.
  - 기본 프로필
  - 누적 성적
  - 최근 출장
  - 면담 코멘트
  - 코치 리포트
- selector는 hidden stat 숫자 원본을 절대 노출하지 않는다.

## 13. File Structure Guidance

예상 구조:

```text
app/games/_components/baseball-manager/
  BaseballManagerGame.tsx
  BaseballManagerGame.module.css
  constants.ts
  types.ts
  engine/
  sim/
  data/
  ui/
```

권장 역할:

- `engine/`: 상태 전이, 액션 처리, 저장/로드
- `sim/`: 생성형 리그, 경기 계산, 성장/피로 처리
- `data/`: 이름 풀, 팀 생성 규칙, 초기 밸런스 상수
- `ui/`: 패널 컴포넌트, SVG 필드, 로그 뷰

## 14. Test Plan

다음 시나리오는 v1 검증 기준으로 문서에 포함한다.

1. 시즌을 1일 진행하면 일정, 체력 회복, 출장 이력, 2군 성장 처리가 일관되게 반영된다.
2. hidden stat는 어떤 UI 경로로도 숫자로 직접 노출되지 않는다.
3. 동일한 seed와 동일 입력이면 생성 리그와 경기 결과가 재현 가능하다.
4. 투수 혹사 시 체력 저하와 부상/영구 하락 위험이 누적된다.
5. 선수 면담은 mental 성향에 따라 왜곡된 피드백을 반환한다.
6. 코치 능력 차이에 따라 리포트 정확도와 성장 보정이 달라진다.
7. 경기 화면에서 교체/작전 개입 후 전광판과 로그가 즉시 동기화된다.
8. 저장 후 재로드 시 날짜, 로스터, 시즌 기록, 코치 리포트 상태가 유지된다.

## 15. Assumptions

- v1은 시각 연출보다 정보 해석과 운영 판단이 우선이다.
- 경기 엔진은 내부적으로 투구 단위로 설계하지만, 초기 UI는 이를 전부 수동 조작하지 않는다.
- 리그 구조는 KBO풍 운영 리듬을 참고하되 실존 데이터는 사용하지 않는다.
